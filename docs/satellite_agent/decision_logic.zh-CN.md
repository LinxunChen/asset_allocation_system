# 决策逻辑总览

本文档完整说明当前 `satellite_agent` 的决策链路。目标不是只解释“分数怎么算”，而是解释：

- 系统如何收集资讯
- 如何把资讯转成候选机会
- 为什么某些标的会进入 `预备池`
- 为什么某些标的会升级到 `确认池`
- 为什么有些卡片会被推送到飞书，而另一些不会
- 当前系统在哪些地方可靠，哪些地方还存在明显边界

如果你要判断这套系统是否符合你的交易习惯，建议按本文档从上到下看，而不是只看某一张卡片的分数。

## 一、系统目标

当前版本不是一个“全自动交易系统”，而是一个：

`事件驱动 + 技术确认 + 题材联动 + 分层提醒`

的机会挖掘系统。

它更适合帮助你完成这类动作：

- 更早认识值得研究的标的
- 提前形成观察名单
- 在催化落地后更快确认是否值得加仓
- 在大量资讯里只收到少量精选提醒

它当前不直接帮你做的事：

- 自动下单
- 自动仓位管理
- 自动卖出执行
- 用机器学习黑盒完全取代人的判断

## 二、当前三池框架

系统现在的核心结构是三层，但实现完整度不完全相同：

1. `预备池`
   当前已实现第一版。
   作用是提前识别“值得观察或轻仓试探”的标的。

2. `确认池`
   当前已实现，且是主提醒池。
   作用是在正式催化出现后，给出可操作的确认机会卡。

3. `兑现池`
   当前还没有正式落地。
   未来用于减仓、止盈、退出提醒。

所以更准确地说，当前系统是：

`预备池初版 + 确认池主链路 + 兑现池待开发`

## 三、整体链路

一轮运行内，系统的主要流程如下：

1. 读取正式配置和观察池  
2. 自动同步 `watchlist` 到数据库  
3. 并发抓取资讯源  
4. 对抓回的候选事件做筛选和预算截断  
5. 标准化事件并做去重  
6. 抽取事件含义  
7. 拉行情快照  
8. 生成确认池卡片  
9. 生成预备池候选  
10. 做题材链路联动和预备池升池  
11. 精选并发送正式提醒  
12. 可选发送少量预备池轻推  
13. 落盘 `md/json` 复盘

这条链路的主控制逻辑在 [service.py](/Users/linxun/CodeSpace/asset_allocation_system/src/satellite_agent/service.py) 的 `run_once()`。

## 四、观察池与标的范围

系统当前只会围绕观察池中的标的工作，不会任意扫描全市场。

正式配置文件是：

- [agent.json](/Users/linxun/CodeSpace/asset_allocation_system/config/satellite_agent/agent.json)

内部会把观察池拆成：

- 股票组
- ETF 组
- 题材分组

这些分组一方面决定“哪些标的会被扫描”，另一方面决定：

- `题材链路`
- `题材热度`
- `预备池优先级`
- `确认池联动加分`

也就是说，`watchlist` 现在不是单纯的 symbol 列表，而是决策层的一部分。

## 五、资讯收集逻辑

### 1. 当前启用的资讯源

当前主资讯源是：

- `SEC Edgar`
- `Google News`

其中：

- `SEC` 只用于适合 Edgar 的股票
- `Google News` 用于更广泛的观察池抓取

### 2. 抓取方式

Google News 不是全市场扫新闻再归因，而是：

`按观察池 symbol 去构造查询 -> 获取 RSS -> 再做相关性过滤`

所以系统本质上是：

`watchlist-driven`

而不是：

`market-wide crawler`

### 3. 资讯源可靠性现状

资讯源现在做了这些容错：

- 并发抓取
- fetch probe 替代重复健康检查
- 部分源允许健康检查失败后继续抓取
- 单个 symbol 抓取失败不一定拖垮整轮

但目前仍然存在的边界：

- `Google News` 会偶发超时或 `EOF`
- 部分 symbol 会发生 per-symbol timeout
- 有些轮次抓回的候选本身就很少

所以当前系统的资讯层已经“可用”，但还没有达到“强鲁棒”的程度。

## 六、候选事件筛选与深度处理预算

系统不是把抓回来的每一条资讯都完整处理。

当前有两层预算：

1. `max_events_per_run`
   每轮候选上限
2. `max_deep_events_per_run`
   每轮深度处理上限

### 1. 候选优先级怎么排

系统会优先保留更强的催化事件，比如：

- `earnings`
- `guidance`
- `merger / acquisition`
- `strategic partnership`
- `investment / stake / funding`
- `SEC filing`

同时会压低低信号文章，例如：

- `stock price today`
- `top 10`
- `should you buy`
- `history`
- `battle royale`

### 2. 题材代表保底

在高负载时，系统不会只保留单票分数最高的事件。
现在还会给热门题材保留代表事件，避免：

- 热题材里只有单点龙头留下
- 二三线跟随机会在深度处理前就被截断

这一步的目标是让系统更符合你“提前知道机会”的使用场景。

## 七、去重逻辑

系统当前有多层去重：

1. `event_id` 去重  
2. `event_fingerprint` 跨源去重  
3. 观察池过滤  
4. 提醒去重和冷却时间  

这意味着：

- 同一条新闻不应被多次处理
- 同一逻辑的跨源复述会被压掉
- 同一标的短时间不会被重复提醒轰炸

## 八、事件理解逻辑

### 1. 当前主实现

当前默认还是：

`RuleBasedExtractor`

也就是规则抽取器，而不是 LLM。

它会从标题和摘要里判断：

- 事件类型
- 情绪方向
- 重要性
- 新颖度
- 题材相关度
- 风险提示

### 2. 为什么有时会误判

规则抽取的优点：

- 快
- 稳
- 便宜
- 可解释

缺点：

- 复杂语义理解能力有限
- 对“战略合作 / 大厂投资 / 资本加持 / 题材升级”这类事件的理解不如 LLM
- 对泛财经垃圾文的识别也存在边界

所以当前系统的事件理解已经能工作，但不是最终形态。

## 九、行情快照逻辑

系统在确认池和预备池里都会用到行情快照。

### 1. 当前行情层架构

现在不是单依赖 Yahoo，而是：

- `Yahoo` 主源
- `Stooq` 日线备源
- 旧缓存兜底

### 2. 行情快照会提供什么

主要包括：

- `last_price`
- `rsi_14`
- `atr_14`
- `sma_20 / sma_60`
- `relative_volume`
- `support_20 / resistance_20`
- `support_60 / resistance_60`
- `trend_state`
- `is_pullback`
- `intraday_breakout`

### 3. 为什么有时还会失败

主要原因有两类：

1. 第三方行情源临时失败  
2. 某些 symbol 在某些源上的格式兼容问题

当前已经补了：

- `BRK.B -> BRK-B` 这类 symbol 兼容
- 旧快照兜底

所以“资讯抓到了但行情失败”这类问题已经比之前少了，但仍不是 0。

## 十、确认池逻辑

确认池是当前真正负责飞书正式提醒的主池。

一张确认池卡片的生成依赖两个层面：

1. 事件层  
2. 市场层

### 1. 事件层

回答的是：

`这条消息本身值不值得重视？`

核心分数：

- `event_score`

详见：

- [scoring_guide.zh-CN.md](/Users/linxun/CodeSpace/asset_allocation_system/docs/satellite_agent/scoring_guide.zh-CN.md)

### 2. 市场层

回答的是：

`当前价格位置和技术状态能不能做？`

核心分数：

- `market_score`

### 3. 综合层

最终得到：

- `final_score`
- `priority`

然后生成：

- 入场区间
- 止盈区间
- 失效价

### 4. 当前确认池更偏向什么

当前确认池本质上是：

`事件驱动 + 技术确认`

也就是说，系统不是纯图形交易，也不是纯新闻聚合，而是要求：

- 事件足够强
- 市场状态不太差

才更容易进入正式提醒。

## 十一、预备池逻辑

预备池是为了更符合你的交易习惯而加入的：

`先认识 -> 先观察/轻仓 -> 再确认/加仓`

### 1. 预备池看什么

当前预备池主要看：

- 趋势结构
- 相对量能
- 突破预热
- 回踩结构
- RSI 是否过热
- 波动率是否可接受

它当前更偏：

`技术结构预热池`

### 2. 预备池当前输出什么

每个候选会给出：

- `score`
- `setup_type`
- `headline_summary`
- `action_hint`
- `reason_to_watch`

并写入 `md` 的 `预备池` 区块。

### 3. 预备池轻推

当前预备池默认不是正式飞书卡片。
只有在分数足够高时，才会发送极少量轻推：

- 默认每轮最多 `1-2` 条

这样做是为了避免你被“还没确认的票”刷屏。

## 十二、预备池 -> 确认池 升池逻辑

这是当前系统最贴近你交易习惯的一层。

### 1. 升池发生条件

如果某只票：

- 近期已经进入过预备池
- 本轮又出现了正式事件
- 且事件分数达到一定阈值

那么系统会把它视作：

`预备池候选获得确认`

### 2. 升池效果

升池后，系统会：

- 在卡片上标记 `promoted_from_prewatch`
- 记录 `prewatch_score`
- 增强 `positioning_hint`
- 给最终分数一个小幅确认加成

这意味着：

- 这只票不是“第一次出现”
- 而是“之前已经值得观察，现在获得了正式确认”

这和你“先轻仓，后确认再加仓”的习惯是匹配的。

## 十三、题材链路逻辑

题材链路一开始只是展示层，现在已经进入决策层。

### 1. 当前题材链路做了什么

它会识别：

- 某个 symbol 属于哪些题材组
- 同题材是否已有确认标的
- 同题材是否还有预备池候选

### 2. 当前题材链路在哪些地方影响决策

1. 候选事件深度处理预算  
   热题材更容易保住代表事件

2. 预备池扫描优先级  
   热门题材会被优先扫描

3. 预备池近阈值补录  
   同题材已有确认标的时，近阈值票更容易进入预备池

4. 预备池题材活跃记忆  
   最近几轮持续升温的题材，会继续抬高同题材预备池候选的出现概率

5. 确认池排序和说明  
   最终卡片里会注明：
   - 同题材已有确认标的
   - 题材联动正在形成

### 3. 这一步的意义

它让系统不再只是：

`单票新闻 -> 单票打分`

而更接近：

`题材升温 -> 预热候选 -> 正式确认 -> 链条扩散`

## 十四、飞书正式提醒逻辑

确认池卡片不是生成就发。

系统现在会再做一层精选：

- `priority` 判断
- `normal_alert_min_final_score`
- 每轮提醒总量限制
- 每个 symbol 的提醒上限
- 去重冷却

所以：

- 有卡片
  不等于
- 一定会发到飞书

### 1. 当前正式提醒更像什么

它更像：

`本轮精选后的正式机会卡`

而不是：

`抓到一条新闻就立刻弹消息`

### 2. 这样做的好处

- 降低噪声
- 保留上下文
- 符合你“只看少量高价值机会”的目标

## 十五、md 复盘是怎么组织的

当前 `latest_live_review.md` 已经不只是流水日志，而是结构化复盘。

主要区块包括：

- 结论摘要
- 关键指标
- 运行健康
- 机会概览
- 提醒结果
- 最终推送卡片
- 升池确认
- 题材链路
- 预备池
- 预备池轻推
- 数据源状态
- 卡片解读

所以：

- 飞书负责“即时提醒”
- md 负责“完整复盘”

## 十六、当前可靠性边界

这部分很重要，因为它决定你该如何信任这套系统。

### 1. 目前较可靠的部分

- 观察池驱动的扫描机制
- 事件入库、去重、限流
- 多源行情兜底
- 正式卡片生成与飞书发送
- 题材链路加分和展示
- 预备池与确认池升池链路

### 2. 目前仍然需要保留怀疑的部分

- Google News 的稳定性仍不是企业级
- 规则抽取器对复杂语义仍然有限
- 预备池更偏技术预热，还不够“认知预热”
- 题材热度记忆刚接入，仍需更多 live run 验证
- 兑现池还没落地，仓位管理还不完整

所以当前最准确的评价不是：

- “这已经完全可靠”

而是：

- “这已经是可用的一代产品，但仍需要继续做稳和做深”

## 十七、当前最值得优化的空间

如果你在看完本文档后要判断“下一步该优先做什么”，我建议优先看这几类问题：

### 1. 资讯理解升级

当前规则抽取能工作，但不是终局。  
后续接入 LLM 最合适的位置，是：

- 事件语义理解
- 噪声过滤
- 题材影响判断

而不是直接接管仓位和定价。

### 2. 题材链路继续增强

下一步可以继续做：

- 题材热度对深度预算更强的穿透
- 更明确的“补涨 / 扩散 / 轮动”识别

### 3. 预备池继续升级

当前预备池更偏结构预热。  
以后可以继续引入：

- 题材热度记忆的更长周期建模
- 预期事件日历
- 舆情升温

### 4. 兑现池

未来真正完整的闭环应该是：

- 预备池
- 确认池
- 兑现池

而不是只有买入，没有退出。

## 十八、如何用这份文档审视系统

如果你想判断这套系统是否符合自己的交易方法，可以重点问自己这几件事：

1. 我是否接受它当前是“观察池驱动”，而不是全市场发现器？
2. 我是否认可“预备池轻仓观察，确认池再加仓”的结构？
3. 我是否认同当前题材分组本身？
4. 我是否认同当前对强催化与弱新闻的区分方式？
5. 我是否接受当前飞书是“精选确认提醒”，不是“全量情报流”？
6. 我是否觉得预备池现在还不够早，或不够活？
7. 我是否愿意后续把 LLM 接到事件理解层？

如果这些问题里有任何一项你不满意，那就很适合作为下一轮优化的明确入口。

## 十九、一句话总结

当前系统的真实形态，可以概括成：

`观察池驱动的事件挖掘系统`
`= 资讯抓取 + 规则理解 + 技术确认 + 题材联动 + 分层提醒`

它已经可以帮助你：

- 更早认识值得跟踪的票
- 在正式催化出现时更快确认
- 用飞书接收少量精选机会卡

但它还不是终局。

它下一步最有价值的演进方向，大致是：

- 更强的题材联动
- 更稳的资讯层
- LLM 事件理解
- 最终补齐兑现池
